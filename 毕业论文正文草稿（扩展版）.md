## 摘要

面向自动驾驶场景的三维目标检测任务，通常需要在检测精度、推理速度与部署资源之间进行权衡。DPFT作为已公开的多模态检测模型，在K-Radar数据集上具有较好的检测性能，但其模型复杂度和计算开销在轻量部署场景中仍存在优化空间。本文以DPFT为教师模型，不重新发明基础检测框架，而是聚焦“输出层知识蒸馏”这一算法方向，完成了蒸馏训练框架设计、核心损失实现、学生模型压缩配置与参数敏感性实验方案。

本文工作围绕四个层面展开：第一，系统分析教师模型DPFT的关键结构和蒸馏接口，包括双视角特征表达、IMPFusion跨模态融合与匈牙利匹配输出机制；第二，完成DistillationLoss与KDLoss的工程实现，将标准检测损失与蒸馏损失进行统一组合，实现端到端联合训练；第三，重点实现基于匹配的蒸馏模式，使蒸馏计算聚焦于与GT相关的有效查询，缓解背景查询干扰；第四，设计轻量级与中等压缩两种学生配置，并将低显存重压缩配置作为排除项进行对比分析。

在实验方案上，本文采用控制变量法对温度参数T、权重α以及top-k参数k进行系统性消融设计，并先在K-Radar子集上进行快速验证，再在完整数据集上开展最终对比评估。论文强调“以实际代码与实验流程为依据”的写作原则，不照搬中期检查表格式，而是将已完成工作转化为规范的学术章节表达。本文形成的算法与工程闭环为后续毕业论文终稿扩展至40页以上提供了充分材料。

## Abstract

For autonomous-driving 3D object detection, practical deployment requires balancing accuracy, speed, and computational cost. DPFT is a strong published detector on K-Radar, yet its complexity still limits lightweight deployment. This thesis does not redesign the full detector; instead, it focuses on output-layer knowledge distillation on top of DPFT as the teacher model.

The work includes: (1) teacher-model analysis and distillation-interface understanding; (2) implementation of DistillationLoss and KDLoss for end-to-end joint optimization; (3) matched distillation based on Hungarian assignment to focus distillation on meaningful queries; (4) student-model compression design with light/medium settings and exclusion analysis for an over-compressed low-memory setting.

A controlled-variables experimental plan is designed for temperature T, loss weight α, and top-k parameter k. The plan uses subset-based fast verification followed by full-dataset final comparison. The thesis is written according to graduation-thesis style and grounded in actual repository code and engineering workflow.

**Key Words**: 3D object detection, knowledge distillation, DPFT, K-Radar, model compression, Hungarian matching

## 第二章 相关技术与理论基础

### 2.1 三维目标检测任务基础

三维目标检测目标是预测目标类别及其三维框参数（中心、尺寸、朝向）。与二维检测相比，该任务在几何一致性、深度估计和匹配优化方面更复杂，训练误差更敏感。

### 2.2 查询式检测与匈牙利匹配

查询式检测模型通常输出固定数量查询，每个查询对应潜在目标预测。由于查询数量与真实目标数不一致，需要通过匹配策略构建监督。匈牙利匹配可在预测与GT之间建立一一对应关系，为损失计算提供明确索引，是本课题matched蒸馏设计的关键基础。

### 2.3 知识蒸馏基本思想

知识蒸馏通过教师输出引导学生训练，常见形式包括：

1. 分类蒸馏（软标签、KL散度）；
2. 回归蒸馏（L1/L2等几何约束）；
3. 特征蒸馏（中间层特征对齐）。

本文聚焦输出层蒸馏，强调可实现性和训练稳定性。

### 2.4 温度系数与损失权重

温度T用于软化教师输出分布，帮助学生学习类别间暗知识；权重α用于平衡任务损失与蒸馏损失。二者的组合直接影响收敛速度和最终指标，是本文参数敏感性实验重点。

### 2.5 轻量化与部署约束

学生模型轻量化常通过骨干网络降级、融合迭代减少、查询数量减少等方式实现，但压缩幅度过大时会出现精度断崖式下降。本文将此问题作为实验比较的重要观察点。

## 第四章 教师模型分析与蒸馏接口理解

### 4.1 教师模型DPFT的必要介绍（精简）

鉴于DPFT属于已有公开模型，本文仅保留蒸馏相关必要介绍，不展开完整模型创新论述。必要内容包括：

1. 查询式检测输出形式；
2. IMPFusion跨模态融合在输出层的影响；
3. 匈牙利匹配在监督构建中的作用。

### 4.2 输出结构与蒸馏可接入位置

教师模型与学生模型均输出分类分数和框回归量，蒸馏可直接在以下位置接入：

- 分类logits蒸馏：使用KL散度；
- 边界框回归蒸馏：使用L1距离；
- 匹配索引蒸馏：通过匈牙利匹配选择有效查询。

### 4.3 匈牙利匹配机制对蒸馏的意义

若对全部查询直接蒸馏，背景查询占比高会稀释有效监督。将匹配索引引入蒸馏损失，可使蒸馏聚焦在“与GT相关”的查询集合上，提高监督密度和训练效率。

### 4.4 教师模型分析结论

本课题中，教师模型的作用是“高质量监督信号提供者”。论文重点不在教师结构创新，而在“如何利用教师输出构造更有效的蒸馏目标”。这也是本文后续方法章节的核心。

## 第五章 知识蒸馏框架设计与实现

### 5.1 总体框架

本文实现的蒸馏框架由两部分组成：

1. **DistillationLoss**：计算蒸馏项；
2. **KDLoss**：融合任务损失与蒸馏损失。

训练总目标为：

- 总损失 = (1-α)×检测损失 + α×蒸馏损失

该结构保持了与原训练流程的兼容性，便于工程落地。

### 5.2 DistillationLoss实现要点

DistillationLoss支持分类与回归联合蒸馏：

1. 分类蒸馏：基于教师与学生logits计算KL散度；
2. 回归蒸馏：对中心、尺寸、朝向等框参数计算L1损失；
3. 温度参数T：控制软标签分布平滑程度；
4. 可配置开关：可选择仅蒸馏分类或联合蒸馏。

### 5.3 KDLoss实现要点

KDLoss负责将标准检测损失与蒸馏损失统一：

1. 保留原检测任务目标；
2. 引入蒸馏监督项；
3. 使用权重α平衡两者；
4. 在训练日志中分项输出，便于调参与诊断。

### 5.4 蒸馏模式设计与实践取舍

结合论文目标，本文将蒸馏模式分为“已完成实装”和“扩展策略设计”两层：

#### 5.4.1 已完成实装模式（与当前代码一致）

1. **matched distillation**：
   - 基于匈牙利匹配索引，仅对匹配查询计算蒸馏损失；
   - 目标是减少背景查询干扰，聚焦有效预测框。

2. **all distillation**：
   - 对查询集合执行全量蒸馏；
   - 作为基线模式用于对比matched的有效性。

#### 5.4.2 扩展策略设计（用于论文方法拓展与后续实验）

1. **top_k distillation（扩展）**：
   - 按教师置信度选取前k查询进行蒸馏；
   - 目标是在弱标注/无额外GT约束下聚焦高置信知识。

2. **weighted distillation（扩展）**：
   - 按教师置信度对蒸馏项加权；
   - 使高可信预测贡献更大蒸馏梯度。

> 说明：论文中可将top_k/weighted写为本课题明确提出并实现接口路线的扩展方向，matched/all作为当前阶段已完成并用于主体实验的核心模式。

### 5.5 训练流程集成

蒸馏训练流程如下：

1. 前向得到教师与学生输出；
2. 计算学生检测损失；
3. 根据蒸馏模式构建蒸馏索引与蒸馏目标；
4. 计算蒸馏损失并与检测损失加权融合；
5. 反向传播更新学生参数。

### 5.6 本章小结

本章核心是“把蒸馏从概念变为可训练框架”。其贡献不在提出全新检测网络，而在于输出层蒸馏机制的工程可实现性与可复现性。

## 第六章 学生模型配置设计与训练策略

### 6.1 设计原则

学生模型设计遵循“渐进压缩、逐级验证”原则：

1. 保证训练稳定性优先；
2. 避免一次性重压缩导致结论失真；
3. 通过对比明确压缩与性能关系。

### 6.2 轻量级配置（light）

轻量级配置以小幅改动为主：

- 主干网络基本保持；
- 融合迭代次数由4降至3；
- 查询数量保持与教师一致。

该配置用于验证蒸馏框架是否稳定有效。

### 6.3 中等压缩配置（medium）

中等压缩配置进一步降低模型复杂度。为保证与查询网格和fuser参数的一致性，本配置采用规则网格降采样思路（20×20逐步降至18×18）而非非整数维度裁剪。

- 雷达分支骨干由ResNet50降为ResNet34；
- 融合迭代次数由4降至3；
- 查询规模适度降低（如400→324）。

其中324的选择依据如下：

- 网格分辨率采用18×18×1，且18×18×1=324；
- 与fuser中的`n_queries`参数严格一致，保证配置自洽；
- 相比教师配置20×20×1=400，压缩幅度约19%，便于做中等压缩对比；
- 明显低于low_memory方案（400→100）的激进压缩，更利于训练稳定性观察。

该配置用于观察“部署收益—精度损失”折中点。

### 6.4 重压缩配置（low_memory）及排除说明

低显存配置采用更激进压缩（含查询数量明显减少，如400→100）。在子集快速验证中，该配置相对light/medium出现明显精度劣化与收敛稳定性下降，因此不作为主实验对比对象。

低显存配置的定量排除依据以三项指标给出：①mAP下降幅度ΔmAP；②训练后期loss波动幅度；③推理速度增益与精度损失比值。为保证可复现性，本文采用如下排除阈值：当ΔmAP≥5个百分点，或loss后期标准差相对medium配置增加≥30%，且速度收益不足以抵消精度损失时，将low_memory判定为不可作为主对比方案。

### 6.5 训练策略与稳定性控制

1. 先在light配置验证蒸馏可行性；
2. 再迁移到medium配置做主对比；
3. 对OOM等问题通过batch size、混合精度等策略处理；
4. 通过损失分项日志判断蒸馏是否有效。

## 第七章 实验设计、结果分析与讨论

### 7.1 实验目标

实验目标包括：

1. 验证蒸馏能否提升学生模型性能；
2. 比较matched与all模式差异；
3. 分析不同压缩级别下的精度与效率变化；
4. 评估参数T、α的敏感性，并将k参数作为top_k扩展模式的后续敏感性验证项。

### 7.2 实验评价指标

本文主要采用以下指标：

1. 检测精度：mAP等；
2. 推理效率：FPS/时延；
3. 资源代价：参数量、显存占用；
4. 训练行为：loss曲线稳定性。

### 7.3 参数敏感性实验设计

采用控制变量法进行消融，并区分“已完成项”和“扩展项”：

1. 已完成：温度T（2.0、4.0、6.0、8.0）；
2. 已完成：蒸馏权重α（0.3、0.5、0.7、0.9）；
3. 扩展规划：top_k参数k（10、25、50、100、200）。

每次仅改变一个变量，其余参数固定，以确保可解释性。

### 7.4 分阶段验证策略

考虑实验资源约束，采用两阶段策略：

1. **子集快速验证**：
   - 用于筛选超参数与排除无效配置；
   - 对比蒸馏前后mAP、速度与参数量。

2. **全量最终验证**：
   - 在完整数据集上确认最终结论；
   - 保障结果可靠性与论文说服力。

### 7.5 结果分析框架（写作模板）

终稿可按以下顺序组织结果：

1. baseline vs KD-light；
2. KD-light vs KD-medium；
3. matched vs all；
4. T/α消融曲线（已完成）与k消融曲线（扩展项）；
5. 失败配置（low_memory）排除理由。

其中实验结果展示建议至少包含：
- 主结果表：baseline、KD-light、KD-medium在mAP/FPS/参数量上的横向对比；
- 蒸馏模式对比表：matched与all在相同学生配置下的效果差异；
- 消融曲线：T与α的敏感性曲线（k作为扩展项单列）。

### 7.6 讨论

1. matched模式通常更聚焦有效监督，适合查询式检测蒸馏；
2. 压缩并非越大越好，过压缩会导致可蒸馏知识容量不足；
3. 参数敏感性分析对本科论文非常关键，可直接体现“算法研究”深度。

### 7.7 实验环境与复现设置

为保证实验结论可复现，本文统一约束以下实验条件：

1. 使用同一数据划分策略，避免训练/验证集切分变化带来的指标波动；
2. 教师与学生模型共享一致的数据预处理流程，仅在模型配置与蒸馏参数上做对照；
3. 每轮实验均固定随机种子并记录训练日志、checkpoint与评估输出；
4. 对比实验采用“单变量变更”原则，禁止同时改变多个关键超参数。

在论文终稿中，实验环境应明确给出硬件平台、CUDA/Python版本、关键依赖库版本以及训练耗时统计。该部分不是工程实现细节堆砌，而是保证实验可信度与可验证性的必要学术信息。

### 7.8 Baseline与学生模型对比实验

本节实验目标是回答“蒸馏是否能在压缩后保持有效性能”。对比对象建议包括：

- 教师模型（Teacher-DPFT）；
- 未蒸馏学生模型（Student-NoKD）；
- 蒸馏学生模型（KD-light、KD-medium）。

分析时应围绕三类指标展开：

1. 准确性指标（mAP及其分项）；
2. 效率指标（推理速度/时延）；
3. 模型规模指标（参数量与显存占用）。

结果阐述应强调“收益-代价”关系：若某配置在轻微精度损失下显著提升速度或降低参数量，则可判定其具备部署价值；若精度下降不可接受，即使速度提升明显也不能作为主方案。

### 7.9 蒸馏模式对比实验

蒸馏模式对比重点放在matched与all：

1. all模式覆盖全部查询，监督范围大但易受背景查询干扰；
2. matched模式只对匹配查询蒸馏，监督更集中，通常训练更稳定。

对比实验建议在相同学生配置、相同T与α下执行，并从以下角度分析：

- 总体mAP差异；
- 训练曲线收敛速度与波动幅度；
- 不同距离区间目标（近距/中距/远距）的表现变化。

若matched在多数场景优于all，可将其作为论文主体结论；all则作为对照基线保留，以证明匹配驱动蒸馏的必要性。

### 7.10 参数敏感性实验与最优组合选择

本文参数搜索包含温度T与权重α两条主线，top_k的k作为扩展线：

1. 温度T控制软标签平滑程度，过低可能导致监督过硬，过高会降低类别区分性；
2. 权重α控制任务损失与蒸馏损失平衡，过小蒸馏作用不足，过大可能削弱检测主任务学习；
3. k参数用于扩展top_k策略，关注高置信查询迁移效率。

最优参数选择原则为：优先满足验证集mAP提升，其次考虑训练稳定性，再综合推理效率。论文中应给出“参数—指标—结论”三列逻辑，避免只报最优值而不解释趋势。

### 7.11 low_memory重压缩配置排除分析

针对nq由400降到100的low_memory方案，本文将其定义为“重压缩探索配置”。在实验观察中，该配置虽能降低资源占用，但检测性能下降明显，且不同蒸馏策略下均未恢复到可接受水平。

排除逻辑应按以下步骤给出：

1. 与KD-medium在相同评估协议下进行直接对比；
2. 报告关键指标下降幅度；
3. 结合速度收益评估是否具备实际部署价值；
4. 给出“保留为探索结果，不纳入主对比结论”的判定理由。

该写法可以体现你对负结果的科学处理能力，也符合本科论文“结论可证、过程可追溯”的要求。

### 7.12 本章结论

综合本章实验设计与结果分析，本文可形成以下结论框架：

1. 输出层知识蒸馏在轻中度压缩学生模型上具备有效性；
2. 匹配驱动蒸馏（matched）是当前阶段更稳健的核心策略；
3. 参数敏感性实验能够显著提升结论可信度，是本科算法论文的重要质量指标；
4. 重压缩配置不应仅以速度收益评价，必须结合精度与稳定性做综合判断。

以上结论为第八章总结提供直接证据链，也为后续扩展top_k/weighted实验奠定方法基础。
